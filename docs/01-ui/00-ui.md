## Requirements

The UI is the operator's primary surface for observing and controlling the framework. It must support every interaction described in PRD §8:

- **DAG visualization** — render the full task DAG, status-aware nodes (in-flight, complete, blocked, needs-human), per-node links to source documents, side panel with task details and live logs.
- **Live log streaming** — per-task stream distinguishing agent reasoning, tool calls, and output artifacts.
- **Document viewer** — section structure rendering, full version history with diffs, the producing-task attribution per version, inline staleness and open-issue indicators.
- **Task control console** — manual task injection with declared resource claims and dependencies, breakpoint insertion, priority override, approval gates for `human_review` task types.
- **Rollback** — revert any document node to a prior version; UI enqueues a recovery task that re-queues downstream tasks (event-log revert + dependency re-queue under the hood, per PRD §8.5).
- **Replay mode** — read-only step-through of a subtree's history: document versions, task executions, agent decisions.
- **Project health dashboard** — open-issue roll-up across the tree, staleness indicators, cumulative token cost by subtree, dependency-impact preview for proposed edits or task injections.

The UI must also surface:

- Lifecycle states from §6.2 (DRAFT, SPEC_REVIEW, APPROVED, IN_PROGRESS, VERIFY, COMPLETE, ISSUE_OPEN).
- Resource-claim state from §6.3 so an operator can see what is currently locked.
- Daemon-triggered tasks (§6.4) distinguishably from operator-injected and agent-generated tasks.

For Phase 1 the UI is single-operator (PRD §13 — no concurrent human editors).

## Design

### Stack

TypeScript end-to-end. Component-driven, statically typed routes and API contracts.

| Concern | Choice | Why |
|---------|--------|-----|
| Language | TypeScript (strict) | Mandated by the user; needed for typed routes and contracts. |
| Framework | React 19 | Largest mature ecosystem for DAG viz, markdown, and primitive libraries. |
| Build | Vite | Local operator tool — SSR adds complexity with no benefit. |
| Router | React Router v7 | Industry-standard router with a decade of production usage and the deepest community and LLM training coverage of any React router. Type-safe params/search-params are layered via `zod` at the boundaries where it matters. |
| Server cache | TanStack Query | Caching, invalidation, and websocket/SSE integration for live updates. |
| Client state | Zustand | Minimal API, fits ephemeral UI state (selection, inspector, palette). |
| Styling | Tailwind CSS v4 | Industry default; integrates cleanly with copy-in primitives. |
| Primitives | shadcn/ui (Radix under the hood) | Copied into the repo — no library lock-in, fully extensible. |
| DAG rendering | React Flow (xyflow) | Purpose-built for this exact use case; custom node types, pan/zoom, minimap. |
| DAG layout | `@dagrejs/dagre` (current) → `elkjs` (planned, pre-`05-task-runner` UI) | Dagre introduced with `02-dag` for the v1 flat doc-tree. Transitioning to ELK before the interactive task DAG ships: dagre is a flat-graph layout — every nested-subtree feature in `02-dag` v1.1/v1.2 (bounds union, depth-based `zIndex`, wrapper `pointer-events`, paint-order ordering) is a workaround for that limit, and the calculus inverts once task-DAG claim grouping, cross-subtree dep edges, and the planned C4-style architecture browser (drill-down across abstraction levels) all want compound-graph primitives. See D10. |
| Markdown | `react-markdown` + `remark-gfm` + `shiki` for code | Standard, extensible, AST-level control. |
| Diffs | TBD (`react-diff-view` vs hand-rolled) | Decided when 01-docs is specified. |
| Package manager | pnpm | Disk-efficient; monorepo-ready if we split. |

### Project layout

Single Vite app for Phase 1. Split into a monorepo only when a separate backend package exists.

```
app/
  src/
    main.tsx
    router.tsx               # createBrowserRouter route config
    routes/                  # route components, one per URL
      Root.tsx               # layout route (AppShell + <Outlet />)
      DagPanel.tsx           # /dag (and / redirects here)
      DocsPanel.tsx          # /docs
      DocViewerPanel.tsx     # /docs/:nodeId
      TaskConsolePanel.tsx   # /tasks
      LogStreamPanel.tsx     # /logs/:taskId
      HealthDashboardPanel.tsx # /health
      ReplayPanel.tsx        # /replay/:subtree
      NotFoundPanel.tsx      # catch-all
    components/
      layout/                # AppShell, Sidebar, Topbar, Inspector, StatusBar
      dag/                   # DAG-specific
      docs/                  # Document viewer
      tasks/                 # Task console
      health/                # Dashboard
      ui/                    # shadcn primitives (copied in)
    lib/
      api.ts                 # typed REST client
      ws.ts                  # websocket / SSE client
      types.ts               # shared types (document, task, lifecycle)
    stores/                  # Zustand stores
    styles/
      globals.css            # Tailwind + theme tokens
  index.html
  vite.config.ts
  tsconfig.json
  package.json
```

### Navigation model

- **Left sidebar** — persistent top-level navigation: DAG (default), Documents, Tasks, Health. Logs and Replay are deep-link only (no sidebar entry).
- **Top bar** — project name, global status chips (queue depth, in-flight count, open-issue count), command-palette trigger.
- **Right-hand inspector** — contextual detail panel shared across routes (DAG node detail, task detail, document version detail). Owned by the shell, not by individual routes.
- **Status bar** — version and connection-state indicator.

### Conventions

- Strict TypeScript: `strict`, `noUncheckedIndexedAccess`. No `any`.
- All API contracts live in `src/lib/types.ts`. The eventual backend imports from the same source.
- No CSS-in-JS. Tailwind utilities + occasional tokens in `globals.css`.
- One Zustand store per concern. No god-store. Server data is never duplicated into Zustand — it lives in TanStack Query.
- Routes own their data fetching via TanStack Query; components are presentational where feasible.

### Build and dev workflow

- `pnpm install`, `pnpm dev`, `pnpm build`, `pnpm typecheck`, `pnpm lint`.
- ESLint with `@typescript-eslint` strict-type-checked preset; Prettier for formatting.
- A passing `pnpm typecheck` and `pnpm lint` are preconditions for any node moving from IN_PROGRESS to VERIFY.

## Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | React 19 over Solid/Svelte | Largest production-grade ecosystem for the specific surfaces we need (React Flow, Radix, markdown). LLM-implementable with high accuracy. |
| D2 | Vite over Next.js | Local operator tool; SSR is dead weight. |
| D3 | React Router v7 | Deepest community, largest LLM training coverage, longest production track record. TanStack Router has better built-in type safety but a much shallower ecosystem; `zod` on params/search-params at the boundaries that need it gets us most of the type-safety benefit without the community cost. |
| D4 | Zustand for client state, TanStack Query for server cache | Separation of ephemeral UI state from cacheable server data prevents the common pattern of mirroring fetched data into Zustand and going stale. |
| D5 | Tailwind v4 + shadcn/ui | We own the primitive surface area; library updates can't break our component API. |
| D6 | React Flow for DAG | Purpose-built. Custom rendering only if perf degrades past ~500 visible nodes — defer. |
| D7 | pnpm | Disk-efficient and monorepo-ready. |
| D8 | Single Vite app for Phase 1 | Defer monorepo split until the backend package exists; premature monorepoing is pure tax. |
| D9 | Single cream theme — no dark mode | Warm off-white surface for long operator sessions; less harsh than pure white. Single theme keeps the token surface minimal, eliminates the on-load flash, and removes theme-switching from every component contract. |
| D10 | Keep React Flow; swap layout engine from `@dagrejs/dagre` to `elkjs` before `05-task-runner` ships interactive task DAG UI. Rejected: Cytoscape.js (replaces React Flow; re-authors every custom node through Cytoscape's stylesheet system, loses React component model across three planned panel types), hand-rolled SVG (rebuilds pan/zoom/viewport for a panel that already works), stay-with-dagre (defensible only if task DAG were also flat — it isn't). | Three forward-looking drivers, all already-named or in-flight: (a) `05-task-runner` task DAG is **interactive** (drag-to-reorder, manual claim reassignment, edge creation), which keeps React Flow structurally cheaper than swapping the whole canvas; (b) the task DAG needs **claim grouping, agent grouping, and cross-subtree dep routing** — compound-graph primitives ELK has natively and dagre approximates only via app-side workarounds (see `02-dag` v1.1/v1.2 patch list); (c) the planned **C4-style architecture browser** (System Context → Container → Component drill-down across separate diagrams) will reuse the same canvas with per-level layout configs — ELK supports per-view algorithm selection (`layered`, `mrtree`, `force`), dagre is one algorithm only. **Migration shape:** dagre → ELK touches `useDagLayout.ts` only; custom node components, theme tokens, inspector flow, pan/zoom, `Controls`, `Background` all preserved. **Bundle cost:** elkjs is ~700 KB (~250 KB gzipped) — mitigated by route-level lazy-load (DAG/tasks/architecture are all non-landing routes) and ELK worker bundle (`elkjs/lib/elk.bundled.js`) keeping the main thread clean. **Supersedes** `02-dag` D2 (dagre-over-elkjs); the v1.0 calculus held while the panel was a flat doc-DAG of ≤30 nodes and is recorded as durable provenance there. **Trigger to execute:** the next `02-dag` v1.X cycle (leaf-workflow §8b), scheduled before `05-task-runner` ships its UI. |

## Open Issues

- **Transport choice (WebSocket vs SSE) for live updates.** Log streams and DAG transitions are append-only — SSE is simpler. Task console needs upstream actions (manual injection, approvals), which the REST layer handles fine. Leaning SSE-only for streams + REST for actions; finalize alongside the API server design. *(Priority: MEDIUM, constrains `src/lib/ws.ts`.)*
- **Diff renderer choice.** `react-diff-view` is heavy but feature-complete; hand-rolling over Tailwind keeps bundle small but is more code. Defer to the document viewer node. *(Priority: LOW, deferred to 01-ui/02-docs when written.)*
- **Empty-state strategy.** Each panel can either render against a static fixture during early development or against a stub API returning empty arrays. Leaning stub API so the shell exercises the real data path. *(Resolved within `01-shell`.)*
- **Command palette scope.** `cmdk` is the obvious choice but the command surface is undefined. Stub the button in the shell; design the command surface in a later node. *(Priority: LOW.)*
